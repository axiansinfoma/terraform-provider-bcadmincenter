# Resource (bcadmincenter_environment)

Manages a Business Central environment in the Admin Center.

This resource creates and manages Business Central environments (Production or Sandbox). Environment creation is an asynchronous operation that can take several minutes to complete.

Manages the lifecycle of Business Central environments (Production or Sandbox). This resource handles:

- Creating new environments with specified configuration
- Monitoring environment creation progress (asynchronous operation)
- Deleting environments when no longer needed
- Importing existing environments into Terraform state

## Important Notes

~> **Warning:** Environment creation is an asynchronous operation that typically takes 15-30 minutes to complete. Ensure your Terraform execution environment can handle long-running operations.

~> **Warning:** Most environment attributes cannot be changed after creation and will force recreation (destroy and recreate) if modified. This includes name, type, country, ring, and application version.

~> **Warning:** Deleting a Business Central environment is **permanent and irreversible**. All data, configurations, and customizations will be lost.

## Example Usage

### Basic Production Environment

```terraform
# Example: Production Environment

terraform {
  required_providers {
    bcadmincenter = {
      source = "vllni/bcadmincenter"
    }
  }
}

provider "bcadmincenter" {
  # Authentication can be configured via environment variables:
  # AZURE_CLIENT_ID
  # AZURE_CLIENT_SECRET
  # AZURE_TENANT_ID
}

resource "bcadmincenter_environment" "production" {
  name               = "production"
  application_family = "BusinessCentral"
  type               = "Production"
  country_code       = "US"
  ring_name          = "Production"
  azure_region       = "westus2"

  timeouts {
    create = "90m"
    delete = "60m"
  }
}

output "web_client_url" {
  value       = bcadmincenter_environment.production.web_client_login_url
  description = "The URL to access the Business Central web client"
}

output "environment_status" {
  value       = bcadmincenter_environment.production.status
  description = "The current status of the environment"
}
```

### Sandbox Environment with Specific Version

```terraform
resource "bcadmincenter_environment" "sandbox" {
  name                = "dev-sandbox"
  application_family  = "BusinessCentral"
  type                = "Sandbox"
  country_code        = "US"
  ring_name           = "PROD"
  application_version = "24.0"
  azure_region        = "eastus"

  timeouts {
    create = "90m"  # Extended timeout for slower regions
    delete = "60m"
  }
}
```

### Using Data Source for Ring Selection

```terraform
data "bcadmincenter_available_applications" "apps" {}

locals {
  # Find US production ring
  us_rings = [
    for country in data.bcadmincenter_available_applications.apps.application_families[0].countries_ring_details :
    country if country.country_code == "US"
  ]
  
  production_ring = [
    for ring in local.us_rings[0].rings :
    ring if ring.production_ring
  ][0]
}

resource "bcadmincenter_environment" "production" {
  name               = "production"
  application_family = data.bcadmincenter_available_applications.apps.application_families[0].name
  type               = "Production"
  country_code       = "US"
  ring_name          = local.production_ring.name
  azure_region       = "westus2"
}
```

### Multiple Environments

```terraform
locals {
  environments = {
    production = {
      type   = "Production"
      region = "westus2"
    }
    staging = {
      type   = "Sandbox"
      region = "westus2"
    }
    development = {
      type   = "Sandbox"
      region = "eastus"
    }
  }
}

resource "bcadmincenter_environment" "environments" {
  for_each = local.environments

  name               = each.key
  application_family = "BusinessCentral"
  type               = each.value.type
  country_code       = "US"
  azure_region       = each.value.region

  timeouts {
    create = "90m"
  }
}

output "environment_urls" {
  value = {
    for name, env in bcadmincenter_environment.environments :
    name => env.web_client_login_url
  }
}
```

## Import

```shell
terraform import bcadmincenter_environment.example "/tenants/9ff11aaa-cddc-4df5-97c9-b9e79db1ba1d/providers/Microsoft.Dynamics365.BusinessCentral/applications/BusinessCentral/environments/production"
```

The import ID format is:
```
/tenants/{tenantId}/providers/Microsoft.Dynamics365.BusinessCentral/applications/{applicationFamily}/environments/{environmentName}
```

Where:
- `{tenantId}` - The Azure AD tenant ID
- `{applicationFamily}` - The application family (e.g., "BusinessCentral")
- `{environmentName}` - The name of the environment

## Timeouts

The `timeouts` block allows you to specify custom timeouts for operations:

```terraform
resource "bcadmincenter_environment" "example" {
  # ... other configuration ...

  timeouts {
    create = "90m"  # Default: 60m
    delete = "60m"  # Default: 60m
  }
}
```

- `create` - Timeout for creating the environment (default: 60 minutes)
- `delete` - Timeout for deleting the environment (default: 60 minutes)

Environment creation involves multiple asynchronous steps:
1. Request validation (1-2 minutes)
2. Resource allocation (5-10 minutes)
3. Environment provisioning (10-20 minutes)
4. Final configuration (2-5 minutes)

If creation consistently times out, consider increasing the timeout or checking the Azure region availability.

## Ring Names

The `ring_name` attribute specifies which release ring to use for the environment. Available values:

- `PROD` - Production ring (default) - Most stable, fully tested releases
- `PREVIEW` - Preview ring - Early access to upcoming features with additional testing
- `FAST` - Fast ring - Earliest access to new features, less testing

The ring determines which Business Central updates and features are available in the environment. Most organizations should use the `PROD` ring for production environments.

## Environment Types

### Production Environments

Production environments are intended for live business operations:

- **Higher resource allocation** - More compute and storage resources
- **Backup and recovery** - Automated backups and point-in-time restore
- **SLA guarantees** - Higher availability SLAs
- **Change management** - More restrictive update policies
- **Capacity limits** - May have tenant-level limits on number of production environments

### Sandbox Environments

Sandbox environments are for development, testing, and training:

- **Isolated from production** - No impact on live operations
- **Flexible configuration** - Easier to modify and experiment
- **Sample data** - Can be created with demonstration data
- **No SLA** - Lower availability guarantees
- **Cost-effective** - Lower resource allocation and costs

## Common Issues

### Environment Creation Timeout

If environment creation times out:

1. Check Azure region availability and health
2. Verify your tenant has available capacity
3. Increase the `create` timeout value
4. Check the Business Central Admin Center for the operation status

### Environment Already Exists

If Terraform reports the environment name is already taken:

1. The environment may exist outside Terraform management
2. Import the existing environment using `terraform import`
3. Or choose a different name

### Permission Denied

If you receive permission errors:

1. Verify your service principal has `AdminCenter.ReadWrite.All` permission
2. Ensure the service principal is member of the AdminAgents group
3. Check that admin consent has been granted for the API permissions
4. Verify you're authenticating against the correct tenant

### Environment Stuck in Creating Status

If an environment remains in "Creating" status:

1. Check the Admin Center for detailed operation status
2. Review Azure service health for the target region
3. Contact Microsoft support if the issue persists
4. Consider deleting and recreating with a different region

## Best Practices

1. **Use timeouts** - Always configure appropriate timeouts for your deployment scenarios
2. **Plan for recreation** - Since most attributes force recreation, plan environment changes carefully
3. **Use data sources** - Query available applications and rings to ensure valid configurations
4. **Name carefully** - Environment names are permanent (within the environment lifecycle)
5. **Region selection** - Choose regions close to your users for best performance
6. **Sandbox first** - Test configurations in sandbox environments before creating production
7. **State management** - Use remote state for team collaboration
8. **Secrets management** - Store Application Insights keys and other secrets securely
9. **Documentation** - Document your environment naming conventions and purposes
10. **Monitoring** - Integrate Application Insights for environment monitoring
